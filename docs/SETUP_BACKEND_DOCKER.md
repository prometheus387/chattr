# Backend Docker containers

The backend needs four backing services. All of them are defined in
`chattr_Backend/docker-compose.yml` and brought up with one command.

## At a glance

| Service | Image | Host port | Container port | Purpose |
| --- | --- | --- | --- | --- |
| `postgres` | `postgres:18.3-trixie` | `5432` | `5432` | Primary database |
| `redis` | `redis:8.6.1-trixie` | `6379` | `6379` | Cache / rate-limit (reserved) |
| `valkey` | `valkey/valkey:9-alpine` | — (internal) | `6379` | Cap's job queue / message bus |
| `cap` | `tiago2/cap:latest` | `3000` (loopback only) | `3000` | Captcha admin / verify API |
| `seq` | `datalust/seq:latest` | `80` (loopback only), `5341` (loopback only) | `80`, `5341` | Structured logs |

Loopback-only services (Cap, Seq) bind to `127.0.0.1` so they aren't
exposed to the LAN. Reach them from your own machine only.

## Prerequisites

- Docker Engine 24+
- `docker compose` (v2 syntax, the `docker compose` with a space, not the legacy `docker-compose`)

```bash
docker --version
docker compose version
```

## .env file

`docker-compose.yml` reads three secrets from a `.env` file in the same
directory. Create it before bringing the stack up:

```bash
cd chattr_Backend
cp .env.example .env   # if you have one, otherwise just create the file
```

The required keys:

```dotenv
# Postgres — used by both docker-compose.yml AND appsettings.Development.json
POSTGRES_PASSWORD=dev-local-pw

# Cap admin API
CAP_ADMIN_KEY=cap-admin-token-rotate-me

# Seq first-run admin password (PBKDF2 hash, not the plain password)
SEQ_FIRSTRUN_ADMINPASSWORDHASH=$argon2id$v=19$m=8192,t=4,p=1$...
```

The Cap `SECRETKEY` and `SITEKEY` are generated by the container on first
start and are not required in `.env` initially — see the [Cap](#cap)
section below for how to retrieve and rotate them.

> **Never commit `.env`.** Add it to `.gitignore` and consider
> `chmod 600 .env` so only your user can read it.

## Start, stop, inspect

```bash
cd chattr_Backend

# Start everything in the background
docker compose up -d

# Show running containers + their health
docker compose ps

# Tail the logs (Ctrl+C to stop)
docker compose logs -f

# Logs for a single service
docker compose logs -f postgres

# Stop without removing data
docker compose down

# Stop AND delete volumes (Postgres data, Seq data, etc.)
docker compose down -v
```

`docker compose down -v` is destructive. It will wipe the database, Seq
event log, and the Valkey cache. Only run it when you want a clean slate.

## Service details

### Postgres

- Image: `postgres:18.3-trixie`
- User/db: `chattr / postgres`
- Volume: `postgres-data:/var/lib/postgresql`
- Healthcheck: `pg_isready -U chattr` every 10s

Connect from the host (e.g. for `psql`):

```bash
docker exec -it chattr_postgres psql -U chattr -d postgres
```

The connection string the API uses:

```
Host=localhost;Port=5432;Database=postgres;Username=chattr;Password=${POSTGRES_PASSWORD}
```

> The first time you bring the stack up, the Postgres init scripts run as
> the `postgres` superuser. Subsequent restarts preserve the data in the
> `postgres-data` volume.

### Redis

- Image: `redis:8.6.1-trixie`
- Volume: none (data is in-memory by default; intentional for dev)
- Healthcheck: `redis-cli ping` every 10s

Currently unused by the .NET API. Reserved for future cache / rate-limit
use. Stop it with `docker compose stop redis` if you want a minimal stack.

### Cap

- Image: `tiago2/cap:latest`
- Bind: `127.0.0.1:3000` only — not reachable from the LAN
- Depends on: `valkey` (healthy)

Cap provides a captcha service the frontend can use on the register /
signin forms. The dashboard is at <http://localhost:3000/ui>.

On first boot, the container auto-generates a `SECRETKEY` and `SITEKEY`
and logs them. To retrieve them:

```bash
docker compose logs cap | grep -i 'sitekey\|secretkey'
```

Then store them in your `.env` and the frontend's `.env.local`:

```dotenv
# chattr_Backend/.env
CAP_SECRETKEY=...
CAP_SITEKEY=...

# chattr_Frontend/.env.local
NEXT_PUBLIC_CAP_SITEKEY=...   # only the sitekey goes to the browser
```

Reset / rotate the keys:

```bash
docker compose exec cap cap admin reset
docker compose restart cap
```

### Valkey

- Image: `valkey/valkey:9-alpine`
- Container name: `cap-valkey`
- Volume: `valkey-data:/data`
- Healthcheck: `valkey-cli ping` every 5s

Valkey is the Redis fork that Cap uses for its background job queue. It
isn't on a host port — only `cap` talks to it over the internal Docker
network. Don't worry about it unless Cap is broken.

### Seq

- Image: `datalust/seq:latest`
- Binds: `127.0.0.1:80` (UI) and `127.0.0.1:5341` (ingestion API)
- Volume: `seq-data:/data`

Structured log viewer. The dashboard is at <http://localhost>. Default
admin username: `admin`. The password is the plaintext of whatever you
hashed to produce `SEQ_FIRSTRUN_ADMINPASSWORDHASH`.

To compute a fresh hash:

```bash
docker run --rm datalust/seq:latest hash-password 'MyNewPassword'
```

Send logs to Seq from the API by setting:

```jsonc
// appsettings.Development.json
{
  "Serilog": {
    "WriteTo": [
      { "Name": "Seq", "Args": { "serverUrl": "http://localhost:5341" } }
    ]
  }
}
```

(Add Serilog + the Seq sink to the API to wire this up. The config is
shown here for reference; nothing in the API currently ships logs to Seq
out of the box.)

## Networking & DNS

All services are attached to the default `bridge` network. Service names
(`postgres`, `redis`, `valkey`, `cap`, `seq`) are resolvable as DNS names
**inside** the network. For example, `cap` reaches Postgres as
`postgres:5432`.

The .NET API, however, runs on the host and reaches the services via
`localhost:<port>`. That's why the API's connection string is
`Host=localhost;Port=5432` and not `Host=postgres;Port=5432`.

If you ever containerize the API too, point the connection string at the
service name (e.g. `Host=postgres;Port=5432`) and use Docker's
internal DNS.

## Volumes

```bash
docker volume ls
```

| Volume | Service | Persists |
| --- | --- | --- |
| `postgres-data` | Postgres | Database files |
| `valkey-data` | Valkey | Cap's job queue state |
| `seq-data` | Seq | Event log + user accounts |
| `seq-data` | Seq | (Same as above) |

Two of the services — `redis` and `cap` — use no volume and start fresh on
every `up`. That's intentional for dev; both are stateless from our
perspective.

## Troubleshooting

**`postgres` won't start with `permission denied` on the volume.**

You're on Linux and the volume was created by a different user. Fix:

```bash
docker compose down -v
docker volume prune
docker compose up -d
```

**`cap` shows "waiting for valkey" forever.**

Valkey's healthcheck is failing. Inspect:

```bash
docker compose logs valkey
docker compose exec valkey valkey-cli ping
```

**API logs say `Failed to determine the https port for redirect`.**

Not a container problem — the API tries to redirect HTTP→HTTPS, but no
HTTPS port is configured in dev. Either run the `https` profile from
`launchSettings.json` or live with the warning. It does not break requests
in dev.

**Port `5432` is already in use on the host.**

Something else (probably a local Postgres install) is binding 5432. Either
stop it or change the host port in `docker-compose.yml`:

```yaml
ports:
  - "55432:5432"   # host:container
```

…and update `ConnectionStrings:DefaultConnection` in
`appsettings.Development.json` to `Port=55432`.

## What's *not* in docker-compose

- The .NET API itself (you `dotnet run` it from the source tree)
- The Next.js frontend (`pnpm dev`)
- An nginx / Caddy reverse proxy for production

In production you'd typically run the API and the frontend as separate
services (or a single combined image with the frontend's static export
served by something like nginx), with the docker-compose services as
backing infrastructure.
